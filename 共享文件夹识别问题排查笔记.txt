================================================================================
  Office Word Add-in 共享文件夹无法识别的排查与修复笔记
================================================================================
  日期: 2026-06-15
  项目: office-word-ai-add-in (https://github.com/liuwinbin/office-word-ai-add-in)
  对比参考: word-GPT-Plus (https://github.com/Kuingsmile/word-GPT-Plus)

================================================================================
  问题现象
================================================================================

  manifest.xml 放入 Word "共享文件夹"（受信任的加载项目录）后，Word 的
  "插入 → 加载项 → 共享文件夹" 列表中不显示该加载项。

  而 word-GPT-Plus 的 manifest.xml 在同一个共享文件夹中可以正常识别。

================================================================================
  根因分析：为什么 Word 静默忽略加载项？
================================================================================

  Word 扫描共享文件夹时，会对每个 .xml 文件进行 XSD Schema 校验。
  校验失败不会弹出任何错误提示，而是直接静默跳过该文件。

  经过逐项对比 word-GPT-Plus（可识别）与本项目（不可识别）的 manifest.xml，
  发现以下 5 个差异，其中第 5 项是致命根因。

  ------------------------------------------------------------------------
  修复项 #1: XML 声明中的 standalone="yes"
  ------------------------------------------------------------------------
  严重程度: 中
  状态: ✅ 已修复

  改前:
    <?xml version="1.0" encoding="UTF-8" standalone="yes"?>

  改后:
    <?xml version="1.0" encoding="UTF-8"?>

  原因:
    standalone="yes" 表示 XML 文档完全独立、不依赖外部 DTD。
    这不是 Office Add-in manifest 的标准写法，某些 Office XML 解析器
    可能因此拒绝解析。word-GPT-Plus 无此属性。

  ------------------------------------------------------------------------
  修复项 #2: 示例 GUID
  ------------------------------------------------------------------------
  严重程度: 低
  状态: ✅ 已修复

  改前:
    <Id>a1b2c3d4-e5f6-7890-abcd-ef1234567890</Id>

  改后:
    <Id>2a929139-573a-4b24-a4e2-b59fe277b693</Id>

  原因:
    使用容易记忆的假 GUID 不专业，虽然不会直接导致识别失败，
    但如果与其他加载项 ID 冲突会有问题。word-GPT-Plus 使用真实 GUID。

  ------------------------------------------------------------------------
  修复项 #3: 缺少 <FunctionFile> 元素
  ------------------------------------------------------------------------
  严重程度: 中
  状态: ✅ 已修复

  改前:
    <DesktopFormFactor>
      <GetStarted>...</GetStarted>
      <ExtensionPoint ...>          <!-- 直接跳到这里 -->

  改后:
    <DesktopFormFactor>
      <GetStarted>...</GetStarted>
      <FunctionFile resid="Commands.Url" />   <!-- 新增 -->
      <ExtensionPoint ...>

  原因:
    FunctionFile 在只有 ShowTaskpane 操作时虽然是"可选"的，
    但某些 Office 版本在共享文件夹扫描时会对 schema 进行严格校验。
    word-GPT-Plus 有这一行，我们的没有。

  ------------------------------------------------------------------------
  修复项 #4: 缺少 Commands.Url 资源定义
  ------------------------------------------------------------------------
  严重程度: 中
  状态: ✅ 已修复

  改前:
    <bt:Urls>
      <bt:Url id="OfficeAI.Taskpane.Url" .../>
      <bt:Url id="GetStarted.LearnMoreUrl" .../>

  改后:
    <bt:Urls>
      <bt:Url id="Commands.Url" DefaultValue="...commands.html"/>   <!-- 新增 -->
      <bt:Url id="OfficeAI.Taskpane.Url" .../>
      <bt:Url id="GetStarted.LearnMoreUrl" .../>

  原因:
    第 3 项的 FunctionFile 引用了 Commands.Url 资源 ID，
    必须在 Resources 中定义对应的 URL。word-GPT-Plus 有此定义。

  ------------------------------------------------------------------------
  修复项 #5: <Group> 缺少必需的 <Icon> 子元素  ← 致命根因！
  ------------------------------------------------------------------------
  严重程度: 🔴 致命
  状态: ✅ 已修复（这是共享文件夹识别失败的根本原因）

  改前:
    <Group id="OfficeAI.Group">
      <Label resid="OfficeAI.Group.Label"/>
      <!-- 没有 Icon！-->
      <Control ...>

  改后:
    <Group id="OfficeAI.Group">
      <Label resid="OfficeAI.Group.Label"/>
      <Icon>                                          <!-- 新增 -->
        <bt:Image size="16" resid="OfficeAI.Icon16"/>
        <bt:Image size="32" resid="OfficeAI.Icon32"/>
        <bt:Image size="80" resid="OfficeAI.Icon80"/>
      </Icon>
      <Control ...>

  原因:
    根据 Office Add-in manifest XSD Schema，当 <Group> 位于
    <ExtensionPoint> → <OfficeTab> 内时，<Icon> 是必需子元素，
    必须包含 16x16、32x32、80x80 三个尺寸的图标引用。

    word-GPT-Plus 的 <Group> 有这个 <Icon>，我们的没有。

    缺少必需元素 → Schema 校验失败 → Word 静默跳过 → 不在列表中显示。

    这就是为什么所有其他问题修复后仍然识别不到——这个才是真正的 root cause。

================================================================================
  为什么 word-GPT-Plus 可以在共享文件夹中识别？
================================================================================

  它的 manifest.xml 结构完整，包含了所有必需元素：
    - 无 standalone 属性
    - 有真实 GUID
    - 有 FunctionFile + Commands.Url
    - <Group> 里有 <Icon>（16/32/80）

================================================================================
  Word 共享文件夹的加载机制
================================================================================

  1. Word 在"受信任的加载项目录"中递归扫描 .xml 文件
  2. 对每个 manifest.xml 进行 XSD Schema 严格校验
  3. 校验通过 → 列出加载项
  4. 校验失败 → 静默跳过（不报错！不提示！）
  5. 用户点击加载项时，才去 SourceLocation 加载实际页面

  关键点:
    - 共享文件夹只做 XML 结构校验，不验证远程 URL 是否可达
    - 图标和 SourceLocation 的 URL 只有在实际加载时才被访问
    - 任何 XSD 校验失败都会导致加载项"凭空消失"，用户无从排查

================================================================================
  排查方法总结
================================================================================

  如果以后遇到类似问题，可以按以下步骤排查：

  1. 找一个已知可以在相同环境识别的 manifest.xml 作为参考模板
  2. 逐元素对比，确保所有必需子元素都存在
  3. 重点检查 <Group> 下的 <Icon>（最容易遗漏）
  4. 检查 XML 声明中是否有非标准属性（如 standalone）
  5. 检查资源 ID (resid) 是否都有对应的 <Resources> 定义
  6. 用真实的 GUID 替换示例 ID

  [2026-06-15] commit (当前) — 重构为单事务混合架构 v1.6
  ─────────────────────────────────────────────────────────
  需求: 完善「AI 语义标注 + Office.js 原生执行」混合架构，
        核心目标：Phase 2 合并为单 Word.run 事务 → Ctrl+Z 一键撤销全部排版。

  架构变更 (v1.5 → v1.6):

    v1.5 问题诊断:
      1. ❌ executeBatches() 每批一个独立 Word.run → Ctrl+Z 只能撤销最后一批
      2. ❌ AI 分析硬截断 MAX_AI_PARAGRAPHS=200 → 大文档后半部分无指令
      3. ⚠️ SMART_FORMAT_SYSTEM_PROMPT ~600 tokens → token 消耗偏高
      4. ⚠️ 无 AI 调用超时保护 → 网络故障时无限等待
      5. ⚠️ generateFallbackInstructions 仅识别标题+正文 → 缺少代码块/列表/引用

    v1.6 解决方案:

    1. 单事务执行（核心变更）:
       旧: Phase 2 = executeBatches() 递归 → N 个 Word.run → N 个 context.sync
       新: Phase 2 = 单 Word.run → 加载全部段落 → applyAll → 清除空行 → 一次 sync
       效果: Ctrl+Z 一键撤销全部排版操作

    2. AI 分批分析:
       - 新增 AI_BATCH_SIZE = 150 段/批
       - ≤150 段: callAIForBatch 单次调用
       - >150 段: 顺序分批 → mergeInstructions 合并
       - 每批含 withTimeout 30s 超时保护
       - 不再截断大文档，全部分析覆盖

    3. AI Prompt 精简（~40% token 节省）:
       - 旧版 ~25 行 / ~600 tokens → 新版 ~15 行 / ~350 tokens
       - 保留 6 种操作定义 + 4 条规则
       - 去除冗余说明和重复约束

    4. 回退方案增强:
       generateFallbackInstructions 新增识别:
         - 代码块: CODE_INDENT_RE + CODE_KEYWORDS 双重检测
         - 列表: BULLET_RE(• - *) + NUMBER_RE(1. 2.)
         - 引用: QUOTE_RE 引号包裹 + 短段落
         - 中英文混排: CJK_RE + hasLatin → addSpaceBetweenCnEn
       - 接受 paragraphTexts 格式（{index, text}）而非 Office.js 对象

    5. 新增基础设施:
       - withTimeout(promise, ms): Promise.race 超时包装
       - callAIForBatch(paragraphTexts, cfg): 单批 AI 调用（含超时 + 解析）
       - mergeInstructions(instructionArrays): 多批指令按 index 合并去重

    6. 删除的代码:
       - executeBatches(): 递归分批执行（~35 行）→ 替换为单事务内联
       - MAX_AI_PARAGRAPHS 常量 → 替换为 AI_BATCH_SIZE

  [2026-06-15] commit (当前) — 重构排版预设按钮：原生格式 + AI 分流 v1.7
  ─────────────────────────────────────────────────────────
  问题: 4 个格式预设按钮（标题加粗居中/首行缩进/段落间距/字号行距）
        全部走 AI 文本重写 + insertHtml 路径。
        结果：AI 返回 <b> 标签 → Word 将其视为正文加粗，
        原有 Heading 样式丢失，标题变为正文格式。

  根因:
    - format_title 等格式预设不需要修改文本内容，只需修改段落样式
    - 旧架构: AI 重写 → HTML 标签 → insertHtml（Heading 样式被丢弃）
    - AI 返回的 <b> + <center> ≠ Word Heading 样式

  解决方案:

    1. 拆分预设为两类:
       ├─ 原生格式预设（4 个）: format_title / format_indent / format_spacing / format_font
       │   └─ 直接通过 Office.js API 操作文档 → 不经过 AI
       └─ AI 内容预设（4 个）: polish / translate_cn2en / translate_en2cn / summarize
           └─ 保持 AI 文本重写路径

    2. 新增原生执行引擎:
       - detectHeadingLevel(text) → 正则匹配返回 1/2/3/0（非标题）
       - executeNativeFormatPreset(presetKey) → Word.run 事务执行
       - applyFormatToParagraphs(paragraphs, presetKey, opts) → 逐段应用
       - 选区优先，无选区回退到全文
